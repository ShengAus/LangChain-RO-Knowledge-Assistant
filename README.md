# LangChain RO Knowledge Assistant

LangChain RO Knowledge Assistant is a small FastAPI application for grounded question answering over local reverse osmosis notes.

The system serves a browser UI at `/`, exposes a `POST /ask` API, retrieves relevant content from markdown files in `data/`, and returns an answer with source files. If an OpenAI-compatible API key is configured, the final answer is generated by the LLM from retrieved context. Without an API key, the app still works by returning a deterministic fallback answer from the matched local text.

## Overview

- FastAPI backend with a browser UI served from the same app
- LangChain document loading and chunking over local markdown files
- Lexical retrieval over chunked notes
- Grounding checks that reject unsupported questions
- OpenAI-compatible LLM integration for answer generation
- Pydantic request and response models

## Architecture

The application uses a single service.

1. The user opens `/` or sends `POST /ask`.
2. Markdown files in `data/` are loaded and split into chunks.
3. The question is normalized before retrieval. For example, `RO` is expanded to `reverse osmosis`.
4. Chunks are ranked by lexical token overlap with the normalized query.
5. A grounding check verifies that the retrieved chunks still overlap with the query.
6. If no grounded match exists, the app returns:
   `I cannot answer from the provided local documents.`
7. If grounded content exists and `OPENAI_API_KEY` is set, the app sends the retrieved context to the LLM.
8. If no API key is set, the app builds a fallback answer from the best matching retrieved paragraph.
9. The response includes the answer and the list of source files.

## Key Files

- `app/main.py`: FastAPI routes for `/`, `/health`, and `POST /ask`
- `app/chains.py`: document loading, retrieval, grounding, fallback logic, and LLM orchestration
- `app/prompts.py`: system prompt used in LLM mode
- `app/config.py`: `.env`-backed settings
- `data/*.md`: local RO knowledge base
- `tests/test_assistant.py`: focused assistant behavior tests

## Setup

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
Copy-Item .env.example .env
```

## Configuration

Settings are loaded from `.env`.

```env
OPENAI_API_KEY=your-api-key
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-4o-mini
APP_HOST=127.0.0.1
APP_PORT=8000
```

`OPENAI_API_KEY` is optional.

- With an API key, the app uses the retrieved context to generate the final answer with the LLM.
- Without an API key, the app returns a fallback answer from the retrieved local notes.

## Run

```powershell
.\.venv\Scripts\python.exe -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

## API

### `GET /`

Serves the browser interface.

### `GET /health`

Returns:

```json
{
  "status": "ok"
}
```

### `POST /ask`

Request:

```json
{
  "question": "What affects RO recovery?"
}
```

Example response:

```json
{
  "answer": "RO recovery is affected by several factors including feed water quality, fouling potential, scaling risk, hydraulic limits, and operating conditions.",
  "sources": ["ro_design_notes.md", "ro_basics.md"]
}
```

PowerShell example:

```powershell
Invoke-RestMethod -Method Post -Uri http://127.0.0.1:8000/ask -ContentType 'application/json' -Body '{
  "question": "What is RO?"
}' | ConvertTo-Json -Depth 5
```

## Retrieval And Grounding

The retrieval path is intentionally simple and local.

- Documents are loaded from `data/`
- Text is split with `RecursiveCharacterTextSplitter`
- Queries are normalized before retrieval
- Chunks are ranked by token overlap with the normalized query
- A grounding gate rejects questions that are not supported by retrieved content
- Acronym-style queries such as `RO` are expanded before scoring

This keeps the logic easy to inspect and deterministic on a small corpus.

## Limitations

- The knowledge base is limited to the markdown files in `data/`
- Retrieval is lexical and optimized for a small local corpus
- The grounding check is intentionally conservative
- Answer quality depends on document coverage and wording
- This is a local prototype, not a production retrieval system
